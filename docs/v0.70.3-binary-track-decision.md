# v0.70.3 -- evaluate the SEA binary track

**Status:** open architectural question, deferred from the v0.70.2 work.

**Context:** v0.70.2 shipped 3 of 5 platform binaries (linux-x64,
darwin-arm64, win32-arm64) via a per-host orchestrator
(`scripts/build-platforms-all.sh`). The win32-x64 build was blocked on
GCP `CPUS_ALL_REGIONS` quota on the `yaw-labs-prod` project, and a
self-inflicted attempt to use the `yaw-build` project hit a 5-hour
timeout on the headless Windows OpenSSH bootstrap. The `yaw-build`
project was deleted afterward. No GCP resources are currently allocated
to the binary track.

This ticket asks: **should the SEA binary track exist in v0.70.3
onward, and if so, what's the right implementation?**

It is not a "delete the binaries" ticket. The product rationale for
them is real (see `BINARY_DISTRIBUTION.md`); the question is whether
the current Node-SEA-on-per-host-SSH implementation is the right way
to deliver that rationale.

## What the binaries are for (the product case)

`BINARY_DISTRIBUTION.md` documents the strategic reason the SEA binary
track was added: **it collapses the install-store classification mess
in `upgrade-cmd.ts`.** Today the upgrade command carries an 8-way
`InstallMethod` union -- `global-npm | pnpm-global | bun-global | npx |
local-node-modules | bundled-app | dev-checkout | unknown` -- and
reverse-engineers which one applies by pattern-matching `argv[1]`
against `/_npx/`, `/app.asar.unpacked/`, `/npm/node_modules/@yawlabs/mcp/`,
and a half-dozen more path markers. With a single binary in the user's
PATH, all of that becomes a no-op: the upgrade is "swap the file" and
`detectInstallMethod()` can return a clean `"binary"` case.

This is documented as a known gap in `BINARY_DISTRIBUTION.md:166-171`:
a binary install currently classifies as `method: "unknown"` and
`upgrade --json` still suggests `npm install -g @yawlabs/mcp@latest`,
which is wrong for a binary. Closing that gap is the whole reason the
binaries exist; without it, the user-facing value is just "I can
double-click an .exe on Windows."

**The npm path does not solve this problem.** It just moves the user
onto one of the 8 install methods (usually `global-npm` or `npx`).
The install-store classification work still has to be done in
`upgrade-cmd.ts`.

## The three real options

### Option 1 -- Drop the binaries, fix the install-store detection

Cut the SEA track entirely. The user base is split across
`npx`/`global-npm`/host-app-bundled installs. `upgrade-cmd.ts` gets a
proper install-method detector that returns 2-3 real cases instead of
8 hypothetical ones, and the `upgrade` command shells out to
`npm update -g @yawlabs/mcp@latest` or `npx -y @yawlabs/mcp@latest`
accordingly.

**Cost:** ~300 lines of build infrastructure removed
(`scripts/build-binary.mjs`, `scripts/build-platform-remote.sh`,
`scripts/build-platforms-all.sh`, `scripts/build-win-x64-ephemeral.sh`,
`scripts/stage-release-asset.mjs`, `sea-config.json`,
`bin/platforms.json`, the SEA section of `release.sh`, the
`--upload-asset` step, the `--build-only` step). ~150 lines added or
reworked in `upgrade-cmd.ts` and the install-method detector. The
Scoop + Homebrew manifests are deprecated; existing installs still
work but no new manifests are published.

**Benefit:** no per-host build infrastructure, no GCP dance, every
release is ~5 minutes. The install-method detection work is real
anyway and has to be done for the `npx`/`global-npm` cases.

**Risk:** the "CLI in my PATH" use case (the one the Scoop + Homebrew
installs serve) loses its dedicated path. `npm i -g @yawlabs/mcp`
is a fine substitute for almost everyone, but a `winget install
yaw-mcp` story is not viable without binaries.

### Option 2 -- Keep the binaries, switch from Node SEA to `pkg`

`@yao-pkg/pkg` (the community fork of Vercel's `pkg`) cross-compiles
between platforms via Docker + QEMU on a single host. The build moves
from "5 hosts + per-host SSH + GCP quota" to "1 workstation + 1
Docker container + `npm i @yao-pkg/pkg`." All 5 platforms build
locally; the GitHub Release is populated from the same machine that
runs the lint + test + publish flow.

**Cost:** rewrite `scripts/build-binary.mjs` against
`@yao-pkg/pkg`'s API (or wrap `npx -y @yao-pkg/pkg`). Drop the
per-host orchestrator entirely. Keep the SHA sidecar + asset-attach
path. Keep the Scoop + Homebrew manifest generation.

**Benefit:** no GCP, no quota, no per-host SSH, no Windows headless
bootstrap. Cross-compile via QEMU works for `linux-x64` ->
`linux-arm64` / `win32-x64` / `darwin-x64` (modulo 5-10x build-time
slowdown per QEMU emulated target). One machine owns the build.

**Risk:** QEMU-emulated builds are slower than native builds (2-5x for
arm64 from x86_64, more for win32 from Linux). Authentication on
macOS-emulated builds may fail (codesign + notarization are not
emulable; you'd sign on a real Mac, post-build, as a separate step).
The macOS-on-Linux path requires `darwin/libc` from a Docker image --
maintained by the `pkg` community, not by us, so build correctness
depends on upstream image freshness.

### Option 3 -- Status quo (Node SEA on per-host SSH)

Keep the current implementation. Provision a real Windows x64 host
(a Windows 11 workstation, an AWS `c6i.xlarge` Windows AMI for ~$0.05/hr,
or wait for the GCP quota bump via Sales). Provision a real arm64
Linux host (Graviton, `tau2t2a`) the same way. The orchestrator
already works; the remaining gap is just the missing hosts.

**Cost:** 1-2 new permanent (or spot) build hosts, $30-50/mo in
ongoing compute for a few-releases-a-month cadence. Windows host
needs OpenSSH + `google_compute_engine` pubkey + per-release user
setup (the `build-win-x64-ephemeral.sh` script in the repo is the
right pattern; it just needs a real host).

**Benefit:** no architectural change. The build is reproducible on
native hosts; no QEMU emulation; the macOS codesign path is native.

**Risk:** the per-host setup is a permanent maintenance burden. The
Windows host is a single point of failure for the `win32-x64` binary.
The `arm64` Linux host is the same for the `linux-arm64` binary. A
host going down (spot reclaim, OS update) blocks that platform's
release until it's restored.

## What this ticket is NOT

- Not a "delete `BINARY_DISTRIBUTION.md`" ticket. The strategic
  rationale it captures is real and the install-store union is real
  work that has to be done regardless of which option we pick.
- Not a "switch to GitHub Actions" ticket. GH Actions was deliberately
  ripped out of the parent Yaw Terminal project and `mcp` followed
  the same shape. Re-introducing it for `mcp` specifically would
  contradict the cross-repo `gh` strip-out we just finished.
- Not a "wait and see" ticket. The empty `bin/platforms.json` is
  actively confusing -- it has `_comment` fields explaining why
  win32-x64 and linux-arm64 are unconfigured, which makes the file
  look like it's awaiting a build, not awaiting a decision. The
  decision needs to land before v0.70.3 ships.

## Recommended next step

Pick an option. If the user base is overwhelmingly host-app-launched
(Claude Desktop, Cursor), Option 1 is the right call -- the binary
path is solving a problem for the 5% of users who want a CLI in their
PATH, and the cost of supporting it is non-trivial. If the user base
includes a meaningful fraction of "I want `yaw-mcp` in my PATH
without a Node install" users (the Scoop + Homebrew install counts
give you a real signal here), Option 2 is the right call -- the
product case stands, the implementation just needs to be simpler.
Option 3 is the right call only if we're already paying for those
hosts and they're not adding a maintenance burden.

The decision is owned by whoever understands the install-base
breakdown. This ticket captures the question, the options, and the
constraints; it does not capture the answer.

## Files in scope for whichever option lands

- `release.sh` -- remove or simplify the `--build-only` /
  `--upload-asset` section (Options 1, 2) or leave as-is (Option 3)
- `scripts/build-binary.mjs` -- replace with `@yao-pkg/pkg` wrapper
  (Option 2) or delete (Option 1)
- `scripts/build-platform-remote.sh` -- delete (Options 1, 2)
- `scripts/build-platforms-all.sh` -- delete (Options 1, 2)
- `scripts/build-win-x64-ephemeral.sh` -- delete (Options 1, 2)
- `scripts/stage-release-asset.mjs` -- keep (Options 2, 3) or
  delete (Option 1)
- `scripts/platforms.json.example` -- delete (Options 1, 2)
- `bin/platforms.json` -- delete (Options 1, 2) or repopulate
  (Option 3)
- `sea-config.json` -- delete (Options 1, 2)
- `src/upgrade-cmd.ts` (or equivalent) -- the install-method
  detector work is the real deliverable, regardless of which option
  is picked. The PoC note in `BINARY_DISTRIBUTION.md:166-171` is the
  spec.
- `BINARY_DISTRIBUTION.md` -- rewrite or delete depending on
  outcome.
- `CHANGELOG.md` -- add a 0.70.3 entry reflecting whichever option
  lands.
